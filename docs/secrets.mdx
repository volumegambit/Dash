Dash handles several kinds of secrets: AI provider API keys, gateway management tokens, chat tokens, and messaging app credentials. How these secrets are stored depends on how you run Dash.

## Gateway tokens

When Mission Control spawns the gateway, it generates management and chat tokens and stores them in the **OS keychain** (macOS Keychain / Windows Credential Manager / libsecret on Linux). These tokens are never written to plaintext files on disk. When MC relaunches, it reads the tokens from the keychain to reconnect to the existing gateway.

## Storage methods

Dash supports three ways to provide secrets, listed from simplest to most secure.

### Environment variables

Set API keys directly in your shell or `.env` file. Good for local development.

```bash
export ANTHROPIC_API_KEY=sk-ant-...
export MANAGEMENT_API_TOKEN=mgmt-...
export CHAT_API_TOKEN=chat-...
```

Environment variables take precedence over `credentials.json` but are overridden by the `--secrets` flag. See [Configuration](/configuration#environment-variables) for the full list.

### Credentials file

A JSON file at `config/credentials.json` or `credentials.json` in your project root. Suitable for single-machine setups where you control file access.

```json
{
  "anthropic": { "apiKey": "sk-ant-..." },
  "openai": { "apiKey": "sk-..." },
  "google": { "apiKey": "AIza..." }
}
```

<Warning>
  This file is not encrypted. Protect it with file permissions (`chmod 600`) and keep it out of version control.
</Warning>

### Encrypted secret store (Mission Control)

Mission Control (desktop app and CLI) stores all secrets in an AES-256-GCM encrypted file. This is the recommended approach for production use.

## Encrypted secret store

All secrets managed by Mission Control live in a single encrypted file:

```
~/.mission-control/secrets.enc
```

This file is created when you first set up Mission Control. It replaces any earlier plaintext `secrets.json` that may have existed.

### Encryption details

| Property | Value |
|----------|-------|
| Algorithm | AES-256-GCM (authenticated encryption) |
| Key derivation | scrypt (N=65536, r=8, p=1) |
| Key length | 256 bits (32 bytes) |
| Salt | 256-bit random, unique per store |
| IV | 96-bit random, unique per write |
| Auth tag | 128-bit GCM tag (integrity check) |
| File permissions | `0600` (owner read/write only) |

<Steps>
  <Step title="You create a password">
    On first use, you choose an encryption password (minimum 8 characters). This is the only secret you need to remember.
  </Step>
  <Step title="A key is derived">
    Your password and a random 256-bit salt are fed into scrypt to produce a 256-bit encryption key. scrypt is intentionally slow (~100ms) to resist brute-force attacks.
  </Step>
  <Step title="Secrets are encrypted">
    The JSON object containing all your secrets is encrypted with AES-256-GCM using the derived key and a fresh 96-bit IV. The GCM authentication tag ensures that any tampering with the ciphertext is detected on decryption.
  </Step>
  <Step title="The key is cached">
    The derived key is stored in your OS keychain so you don't need to re-enter your password on every operation.
  </Step>
</Steps>

### Writes are atomic

Every write to `secrets.enc` goes through a temporary file first. The encrypted payload is written to `secrets.enc.tmp`, then atomically renamed to `secrets.enc`. If the process crashes mid-write, the previous version of the file remains intact. This prevents data loss from interrupted operations.

### Keychain caching

After you enter your password, the derived encryption key is cached in your operating system's credential store. This means you typically enter your password once per session.

| Platform | Backend |
|----------|---------|
| macOS | Keychain (`security` CLI) |
| Linux | Secret Service API (`secret-tool` — works with GNOME Keyring, KDE Wallet) |
| Other | No caching (password prompted each time) |

On macOS, the key is piped to the `security` command via stdin rather than passed as a CLI argument, so it never appears in process listings.

The desktop app uses Electron's `safeStorage` API instead of the OS keychain directly. The derived key is encrypted with the OS-level encryption and stored at `~/.mission-control/session.key`. This file is deleted when you lock the store.

### Lock and unlock

The secret store has two states:

- **Unlocked** — the derived key is in memory and secrets can be read or written
- **Locked** — the key has been cleared and all secret operations require re-authentication

The store auto-unlocks at startup if a cached key is found in the keychain. If the cached key is invalid (e.g., after a password change), it's discarded and you're prompted to unlock manually.

You can lock the store at any time:

<Tabs>
  <Tab title="CLI">
    ```bash
    mc lock      # clear cached key, lock the store
    mc unlock    # re-enter password, cache key
    ```
  </Tab>
  <Tab title="Desktop app">
    Navigate to the Secrets page and click **Lock**.
  </Tab>
</Tabs>

### Changing your password

You can re-encrypt the entire store with a new password without losing any data.

```bash
mc secrets change-password
```

This decrypts all secrets with your current password, generates a new salt, derives a new key from your new password, and re-encrypts everything. The old password stops working immediately.

### Migrating from plaintext

If you have an existing `~/.mission-control/secrets.json` from before encryption was added, it's migrated automatically. The first command that accesses secrets will prompt you to create a password, encrypt the existing data, and delete the plaintext file.

## How secrets reach your agents

When Mission Control deploys an agent, secrets don't stay in the encrypted store — they need to reach the running agent server process. Here's how that handoff works.

<Steps>
  <Step title="Mission Control reads from the encrypted store">
    The relevant secrets (API keys, tokens) are decrypted in memory.
  </Step>
  <Step title="A temporary secrets file is written">
    The secrets are written to a temporary JSON file with `0600` permissions.
  </Step>
  <Step title="The agent server starts with --secrets">
    The agent server process is launched with `--secrets /path/to/temp-file`.
  </Step>
  <Step title="The file is read and deleted">
    The agent server reads the temporary file on startup and deletes it immediately. The secrets exist only in the agent server's memory from that point on.
  </Step>
</Steps>

This approach keeps secrets out of environment variables (which can leak into logs or child processes) and ensures the plaintext secrets file exists on disk for the shortest possible time.

## Credential precedence

When the agent server starts, it resolves credentials in this order (highest priority first):

1. **`--secrets` file** — temporary file from Mission Control
2. **Environment variables** — `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, etc.
3. **`credentials.json`** — plaintext file in the config directory

The first source that provides a value wins. This means a Mission Control deployment always uses the secrets from the encrypted store, regardless of what's in the environment or config files.

## What's stored

Mission Control uses well-known key names for provider credentials:

| Key | Purpose |
|-----|---------|
| `anthropic-api-key` | Anthropic API key |
| `openai-api-key` | OpenAI API key |
| `google-api-key` | Google API key |
| `messaging-app:<id>:token` | Messaging app bot token (Telegram, etc.) |
| `whatsapp-auth:<id>:*` | WhatsApp session credentials |

You can store additional secrets with `mc secrets set <key>`. There's no restriction on key names.

## Security properties

A summary of what the encrypted store protects against and what it doesn't.

**Protected:**
- Reading secrets from disk without the password (AES-256-GCM)
- Tampering with the encrypted file (GCM authentication tag detects modifications)
- Brute-forcing the password (scrypt makes each guess expensive)
- Data loss from crashes during writes (atomic rename)
- Accidental overwrite of an existing store (setup refuses if `secrets.enc` already exists)
- Secrets leaking into process listings (keychain operations use stdin, not CLI args)

**Not protected:**
- An attacker with access to both the encrypted file and the OS keychain (the cached key can decrypt the store)
- Memory-resident secrets in a running agent server process
- A compromised machine with root access

For high-security deployments, consider running agent servers in isolated environments and rotating API keys regularly.
