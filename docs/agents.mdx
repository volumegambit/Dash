An agent is a member of your AI team. It has a model to think with, instructions to follow, and tools to get work done. You tell it what to do, and it figures out how.

## What is an agent?

An agent combines four things:

1. **A model** â€” the AI brain (Claude, GPT, Gemini, etc.)
2. **A system prompt** â€” instructions that shape the agent's personality and behavior
3. **Tools** â€” capabilities like running commands, reading files, or searching the web
4. **Skills** â€” reusable workflows the agent can discover and use on demand

When you send a message, the agent reads it, reasons about what to do, uses tools if needed, and responds. It does this autonomously â€” no hand-holding required.

## Agent configuration

Each agent is defined in your config with a few key fields.

```json
{
  "agents": {
    "assistant": {
      "model": "anthropic/claude-sonnet-4-20250514",
      "systemPrompt": "You are a helpful assistant that specializes in research.",
      "tools": ["bash", "read", "write", "edit", "grep", "glob", "web_search"],
      "maxTokens": 4096,
      "workspace": "./data/assistant-workspace",
      "fallbackModels": ["openai/gpt-4o", "google/gemini-2.0-flash"],
      "skills": {
        "paths": ["./skills"],
        "urls": ["https://example.com/skills"]
      }
    }
  }
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `model` | `string` | yes | The AI model to use (e.g. `anthropic/claude-sonnet-4-20250514`, `openai/gpt-4o`, `google/gemini-2.0-flash`) |
| `systemPrompt` | `string` | yes | Instructions that tell the agent who it is and how to behave |
| `tools` | `string[]` | no | Which tools the agent can use. See [Tools](/tools) for the full list |
| `maxTokens` | `number` | no | Maximum response length in tokens (default: `4096`) |
| `workspace` | `string` | no | Directory where the agent operates, resolved relative to the project root |
| `fallbackModels` | `string[]` | no | Backup models to try if the primary model fails |
| `skills` | `object` | no | Local paths and/or remote URLs where the agent discovers skills |

## Available tools

Pick the tools your agent needs. Each tool gives a specific capability:

| Tool | What it does |
|------|-------------|
| `bash` | Run terminal commands on the system |
| `read` | Read files from the project |
| `write` | Create new files in the project |
| `edit` | Make changes to existing files |
| `ls` | See what files and folders exist |
| `glob` | Find files by name or pattern |
| `grep` | Search for text inside files |
| `web_search` | Search the internet for information |
| `web_fetch` | Download content from web pages |
| `mcp` | Connect to external services and tools |
| `skill` | Use specialized skill workflows |

See [Tools](/tools) for detailed descriptions and sandboxing behavior.

## Fallback models

If your primary model is unavailable or hits a rate limit, the agent automatically tries the next model in the chain.

```json
{
  "model": "anthropic/claude-sonnet-4-20250514",
  "fallbackModels": ["openai/gpt-4o", "google/gemini-2.0-flash"]
}
```

The agent tries models in order: primary first, then each fallback. This keeps your agent running even if one provider has an outage. Each model can be from a different provider â€” just make sure you have the right API keys set up.

## Workspace

Every agent gets its own workspace directory. This is where tool operations happen.

```json
{
  "workspace": "./data/my-agent-workspace"
}
```

The workspace directory is created automatically if it does not exist. Tools operate within this directory:

- **bash** uses the workspace as its working directory
- **read**, **write**, **edit** resolve file paths relative to the workspace
- **File tools** are sandboxed â€” they cannot access files outside the workspace boundary

<Warning>
  If no workspace is configured, tools operate without path restrictions. Always set a workspace for agents that use file or shell tools.
</Warning>

## Lifecycle

Agents are deployed as background processes. Each agent goes through these states:

| State | Meaning |
|-------|---------|
| `provisioning` | The agent is starting up |
| `running` | The agent is live and accepting messages |
| `stopped` | The agent has been stopped (manually or by the system) |
| `error` | Something went wrong during startup or runtime |

You can **start**, **stop**, **restart**, and **remove** agents at any time through Mission Control or the CLI.

<Tabs>

## Sessions

Every conversation with an agent is saved as a session. Sessions are stored as JSONL files (one event per line) so they are easy to inspect and crash-safe.

```
data/sessions/{channelId}/{conversationId}/session.jsonl
```

When you send a new message, the agent loads the session history first. This means it remembers everything from the conversation â€” previous messages, tool results, and its own responses. Context carries across messages automatically.

## Multiple agents

You can run as many agents as you need. Each one is independent â€” different model, different instructions, different tools.

```json
{
  "agents": {
    "researcher": {
      "model": "anthropic/claude-sonnet-4-20250514",
      "systemPrompt": "You are a research assistant. Find information and summarize it clearly.",
      "tools": ["web_search", "web_fetch", "read", "write"]
    },
    "coder": {
      "model": "anthropic/claude-opus-4-20250514",
      "systemPrompt": "You are a senior software engineer. Write clean, tested code.",
      "tools": ["bash", "read", "write", "edit", "glob", "grep"],
      "workspace": "./data/coder-workspace"
    },
    "ops": {
      "model": "openai/gpt-4o",
      "systemPrompt": "You manage infrastructure and deployments.",
      "tools": ["bash", "read"],
      "workspace": "./data/ops-workspace"
    }
  }
}
```

Each agent gets its own workspace, session history, and set of tools. Route messages to the right agent through [Messaging Apps](/messaging-apps) or by name in the [Chat API](/chat-with-your-agent).
