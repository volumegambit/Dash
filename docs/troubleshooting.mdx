Something not working? Here are the most common issues and how to fix them.

## Common issues

<AccordionGroup>
  <Accordion title="Missing API key">
    **Error:**
    ```
    Missing ANTHROPIC_API_KEY. Set it in config/credentials.json or as an env var.
    ```

    **Fix:** Set `ANTHROPIC_API_KEY` in your `.env` file or `config/credentials.json`. See [getting started](/getting-started#configuration).
  </Accordion>

  <Accordion title="Chat connection refused">
    **Symptom:** WebSocket connection to `ws://localhost:9200/ws/chat` is refused.

    **Fix:** Make sure the gateway is running. If using Mission Control, check that the gateway process is healthy from the app. If running standalone, check the server logs for "Gateway channel server listening on port 9200".
  </Accordion>

  <Accordion title="Chat unauthorized (4001)">
    **Symptom:** WebSocket connection closes immediately with code `4001`.

    **Fix:** Include the correct token as a query parameter: `ws://localhost:9200/ws/chat?token=your-chat-token`. Make sure you're using the chat token, not the management token — they are separate.
  </Accordion>

  <Accordion title="Unknown agent error">
    **Error (via Chat API):**
    ```json
    { "type": "error", "id": "...", "error": "Unknown agent: myagent" }
    ```

    **Fix:** The `agent` field in your message references an agent name that doesn't exist in the config. Check `config/dash.json` and make sure the agent name matches a key in the `agents` object.
  </Accordion>

  <Accordion title="Config not found (standalone gateway)">
    **Error:**
    ```
    config/dash.json not found
    ```

    **Fix:** The standalone gateway requires a config file. Pass it explicitly with `--config`:

    ```bash
    npm run gateway -- --config path/to/dash.json
    ```
  </Accordion>
</AccordionGroup>

## Secrets and encryption

<AccordionGroup>
  <Accordion title="Secret store is locked">
    **Error:**
    ```
    Secret store is locked
    ```

    **Fix:** Run `mc unlock` to unlock the store, or any `mc secrets` command — you'll be prompted for your encryption password. If you've previously unlocked in this session and see this error, your keychain cache may have been cleared. Run `mc unlock` again.
  </Accordion>

  <Accordion title="Wrong password / unable to authenticate data">
    **Error:**
    ```
    Unsupported state or unable to authenticate data
    ```

    **Fix:** You entered the wrong encryption password. Try again with the correct password. If you've forgotten your password, the encrypted secrets cannot be recovered — delete `~/.mission-control/secrets.enc` and set up a new password. You'll need to re-enter all your secrets.
  </Accordion>

  <Accordion title="Password must be at least 8 characters">
    **Error:**
    ```
    Password must be at least 8 characters.
    ```

    **Fix:** Choose a longer password. The minimum length is 8 characters.
  </Accordion>

  <Accordion title="Keychain not working / always prompting for password">
    If you're prompted for your password on every command, the OS keychain may not be available. On macOS, make sure the `security` command is accessible. On Linux, make sure `secret-tool` (from `libsecret-tools`) is installed. On unsupported platforms, you'll be prompted each time — this is expected behavior.
  </Accordion>
</AccordionGroup>

## Tool errors

<AccordionGroup>
  <Accordion title="Workspace not found">
    If a tool reports it can't find files, check that the `workspace` path in your agent config exists. Dash creates the workspace directory automatically at startup, but if the config path is wrong, the tool may be looking in the wrong place.
  </Accordion>

  <Accordion title="Path escape blocked">
    **Error:**
    ```
    Error: path "../../etc/passwd" escapes the workspace directory
    ```

    The `read` tool blocks paths that resolve outside the workspace. This is intentional — use absolute paths or paths relative to the workspace root.
  </Accordion>

  <Accordion title="Command timeout">
    **Error:**
    ```
    Command failed
    ```

    The `bash` tool has a 30-second timeout. Long-running commands are killed after this period. Break long operations into smaller steps or increase the timeout in the source code if needed.
  </Accordion>

  <Accordion title="Output truncation">
    The `bash` tool limits output to 100 KB. If a command produces more output, pipe it through `head`, `tail`, or redirect to a file and use `read` to retrieve specific sections.
  </Accordion>
</AccordionGroup>

## Session issues

<AccordionGroup>
  <Accordion title="Corrupt JSONL">
    If a session file has invalid JSON on a line (e.g. from a crash during write), the session loader will throw a parse error. To recover:

    1. Find the session file: `~/.dash/gateway/sessions/{channelId}/{conversationId}/session.jsonl`
    2. Open it in a text editor
    3. Remove or fix the malformed line(s) — each line should be a valid JSON object
    4. Restart Dash

    Since JSONL is append-only, you can safely delete lines from the end without losing earlier conversation history.
  </Accordion>

  <Accordion title="Starting fresh">
    To reset a conversation, delete its session directory:

    ```bash
    rm -rf ~/.dash/gateway/sessions/{channelId}/{conversationId}
    ```

    To reset all sessions:

    ```bash
    rm -rf ~/.dash/gateway/sessions
    ```
  </Accordion>

  <Accordion title="My data moved after updating">
    Dash now keeps everything under `~/.dash` (gateway data, desktop app data, logs, and agent workspaces). The first time a newer version starts, it automatically moves data from the old locations into `~/.dash` — your credentials, agents, and conversations come along.

    If something looks missing, check the old locations:

    - macOS: `~/Library/Application Support/dash` and `~/Library/Application Support/dash-gateway`
    - Linux: `~/.local/share/dash` and `~/.local/share/dash-gateway`
    - Agent workspaces: the old `~/dash-workspaces`

    The migration only runs when `~/.dash` doesn't already have that data, and it never deletes the originals — so it's always safe to copy anything left behind by hand. To pin Dash to a specific location instead, set `DASH_HOME`.
  </Accordion>
</AccordionGroup>

## Docker issues

<AccordionGroup>
  <Accordion title="Data lost on restart">
    Dash stores everything under `DASH_HOME` (set to `/data` in the container). Make sure that path is backed by a persistent volume in `docker-compose.yml`:

    ```yaml
    environment:
      - DASH_HOME=/data
    volumes:
      - dash-data:/data
    ```

    Without this volume, credentials, agents, and sessions live only inside the container and are lost when it's recreated.
  </Accordion>
</AccordionGroup>

## Debug logging

Set `LOG_LEVEL=debug` in your `.env` file or environment for more verbose output:

```env
LOG_LEVEL=debug
```
