Dash is configured with a single JSON file. Define your agent team here — who they are, what models they use, what tools they have access to, and where they can operate.

1. **Built-in defaults** — sensible starting values
2. **JSON config file** — `config/dash.json` or `dash.json` at the project root
3. **Environment variables** — override specific values (via `.env` or shell)

For deployed agents, the `--secrets` flag is the primary way to provide API keys and tokens (see [CLI flags](#cli-flags)). Environment variables can override specific settings.

For Mission Control users, secrets are stored in an encrypted file — see [Secrets](/secrets) for a full explanation of how encryption, key caching, and secret delivery work.

## Config file schema

```json
{
  "agents": {
    "<name>": {
      "model": "anthropic/claude-sonnet-4-20250514",
      "systemPrompt": "You are a helpful assistant.",
      "tools": ["bash", "edit", "write", "read", "glob", "grep", "ls"],
      "maxTokens": 4096,
      "workspace": "./data/workspace",
      "thinking": { "budgetTokens": 10000 }
    }
  },
  "sessions": {
    "dir": "./data/sessions"
  },
  "logging": {
    "level": "info"
  }
}
```

## Agents

Each entry in the `agents` array is one member of your team.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `model` | `string` | yes | Model in `provider/model` format (e.g. `anthropic/claude-sonnet-4-20250514`) |
| `systemPrompt` | `string` | yes | System prompt sent with every request |
| `tools` | `string[]` | no | Tool names to enable. Available: `bash`, `read`, `write`, `edit`, `ls`, `glob`, `grep`, `web_search`, `web_fetch`, `mcp`, `load_skill`, `create_skill`. See [Tools](/tools) |
| `maxTokens` | `number` | no | Maximum response tokens (default: `4096`) |
| `workspace` | `string` | no | Working directory for tools, resolved relative to project root |
| `thinking` | `object` | no | Enable extended thinking. See [extended thinking](/extended-thinking) |
| `thinking.budgetTokens` | `number` | — | Token budget for thinking (must be >= 1024, must be < `maxTokens`) |

### Example: multiple agents

```json
{
  "agents": {
    "default": {
      "model": "anthropic/claude-sonnet-4-20250514",
      "systemPrompt": "You are Dash, a helpful AI assistant.",
      "tools": ["bash", "edit", "write", "read", "glob", "grep", "ls"],
      "maxTokens": 4096,
      "workspace": "./data/workspace"
    },
    "coder": {
      "model": "anthropic/claude-sonnet-4-20250514",
      "systemPrompt": "You are Dash Coder, a software engineering assistant.",
      "tools": ["bash", "edit", "write", "read", "glob", "grep", "ls", "web_search"],
      "maxTokens": 16000,
      "thinking": { "budgetTokens": 10000 },
      "workspace": "./data/coder-workspace"
    }
  }
}
```

### Per-agent files (`agents/` directory)

Instead of defining all agents in `dash.json`, you can create individual JSON files in an `agents/` directory. Each filename (minus `.json`) becomes the agent name.

```
config/
├── dash.json          # sessions, logging (no agents key needed)
└── agents/
    ├── default.json   # → agent named "default"
    └── coder.json     # → agent named "coder"
```

Each agent file contains just the agent config:

```json
{
  "model": "anthropic/claude-sonnet-4-20250514",
  "systemPrompt": "You are Dash Coder.",
  "tools": ["bash", "edit", "write", "read", "glob", "grep", "ls"],
  "maxTokens": 16000
}
```

When an `agents/` directory is present, it **replaces** the `agents` key in `dash.json` entirely. This makes agent definitions portable and easy to share without exposing secrets or session config.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `dir` | `string` | `./data/sessions` | Directory for session JSONL files |

## Logging

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `level` | `string` | `info` | Log level (e.g. `debug`, `info`, `warn`, `error`) |

## Defaults

When no config file is found, Dash uses these defaults:

```json
{
  "agents": {
    "default": {
      "model": "anthropic/claude-sonnet-4-5",
      "systemPrompt": "You are Dash, a helpful AI assistant. You can use tools to help accomplish tasks.",
      "tools": ["bash", "edit", "write", "read", "glob", "grep", "ls"]
    }
  },
  "sessions": { "dir": "./data/sessions" },
  "logging": { "level": "info" }
}
```

## API keys and secrets

API keys are provided through Mission Control's encrypted secret store or the `--secrets` flag when running the gateway standalone.

For local development or standalone use, environment variables can still configure tokens and logging.

## Environment variables

| Variable | Description |
|----------|-------------|
| `MANAGEMENT_API_TOKEN` | Enables the Management API when set |
| `MANAGEMENT_API_PORT` | Management API port (default: `9300`) |
| `CHAT_API_TOKEN` | Enables the Chat API when set |
| `CHAT_API_PORT` | Chat API port (default: `9200`) |
| `LOG_LEVEL` | Log level override |

<Note>
  For details on which models each provider supports and how routing works, see [AI Providers](/ai-providers).
</Note>

## CLI flags

The gateway accepts `--config` and `--secrets` flags for standalone deployment scenarios.

```bash
npm run gateway -- --config /path/to/dash.json --secrets /path/to/secrets.json
```

### `--config <path>`

Load configuration from an explicit path. This can be either:

- **A file** — load that JSON file directly (same as before). If an `agents/` sibling directory exists, agent definitions are loaded from it.
- **A directory** — load `dash.json` from the directory for non-agent settings, and load individual agent definitions from the `agents/` subdirectory.

When `--config` is omitted, the gateway searches for `config/dash.json` or `dash.json` at the project root, and additionally checks `config/agents/` for per-agent files.

### `--secrets <path>`

Load secrets from a JSON file. The file is read once and then **deleted** immediately. This is designed for deployment systems that write a temporary secrets file for the gateway to consume.

```json
{
  "anthropicApiKey": "sk-ant-...",
  "managementToken": "mgmt-...",
  "chatToken": "chat-..."
}
```

When a secrets file is provided, its values are used for provider API keys and auth tokens.

## Encrypted secrets

Mission Control (desktop app and CLI) stores secrets in an AES-256-GCM encrypted file at `~/.mission-control/secrets.enc`. This replaces plaintext storage and protects API keys, tokens, and credentials at rest.

### How it works

1. On first use, you create an encryption password (minimum 8 characters)
2. A 256-bit key is derived from your password using scrypt
3. Secrets are encrypted and written to `secrets.enc` with 0600 file permissions
4. The derived key is cached in your OS keychain (macOS Keychain or Linux `secret-tool`) so you don't need to re-enter your password on every command

### CLI commands

```bash
mc secrets list                  # List all secret key names
mc secrets get <key>             # Show a masked secret value
mc secrets get <key> --reveal    # Show the full value (for scripting)
mc secrets set <key>             # Store a secret (prompts for value)
mc secrets set <key> --value v   # Store a secret inline
mc secrets delete <key>          # Remove a secret
mc secrets change-password       # Re-encrypt with a new password
mc unlock                        # Unlock the store (cache key in keychain)
mc lock                          # Clear the cached key from the keychain
```

### Migration from plaintext

If you have an existing `~/.mission-control/secrets.json` from before encryption was added, it's migrated automatically. The first command that accesses secrets will prompt you to create a password, encrypt the existing data, and delete the plaintext file.

### Credential precedence

When Mission Control manages agents, secrets are delivered to the gateway's encrypted credential store. For standalone gateway usage, credentials are resolved in this order:

1. `--secrets` file (highest priority)
2. Environment variables (`.env` or shell)

## Plugins

Plugins let you drop extra skills into any Dash agent without touching the config file. The gateway discovers plugins from the `plugins/` directory inside your data directory.

### Directory layout

```
<dataDir>/
└── plugins/
    ├── config.json
    └── <plugin-name>/
        ├── .claude-plugin/
        │   └── plugin.json          (optional)
        └── skills/
            └── <skill-name>/
                └── SKILL.md
```

Each plugin follows the Claude Code layout: a `skills/` folder containing one subdirectory per skill, each with a `SKILL.md` file in standard frontmatter format (`name`, `description`, body). The `.claude-plugin/plugin.json` manifest is optional; when present it can set `name`, `displayName`, `description`, and `version`. When absent, the plugin's directory name is used as its name.

### Enabling a plugin

Create or edit `<dataDir>/plugins/config.json`:

```json
{
  "<plugin-name>": { "enabled": true }
}
```

Plugins not listed in `config.json`, or listed with `"enabled": false`, are ignored.

### Example

```bash
# Create the plugin directory and a skill
mkdir -p /path/to/data/plugins/my-skills/skills/summarize

cat > /path/to/data/plugins/my-skills/skills/summarize/SKILL.md <<'EOF'
---
name: summarize
description: Summarize any text concisely
---
Summarize the provided text in three sentences or fewer.
EOF

# Enable the plugin
echo '{ "my-skills": { "enabled": true } }' \
  > /path/to/data/plugins/config.json
```

When the gateway starts, it logs `[plugins] loaded 'my-skills' (skills)` and the agent's `load_skill` tool (see [Tools](/tools) — requires `load_skill` in the agent's tools list) can reference any skill from that plugin.

<Note>
  Only `skills/` is active in this release. Additional plugin components — MCP servers, custom commands, hooks — are coming in a future release.
</Note>

## Deep merge behavior

<Warning>
  JSON config is deep-merged with defaults. Objects are merged recursively, but arrays are **replaced**, not concatenated. Specifying `tools: ["bash"]` in your config replaces the default `["bash", "edit", "write", "read", "glob", "grep", "ls"]` entirely.
</Warning>
