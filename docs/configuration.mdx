title: "Configuration"
description: "Configure agents, sessions, and credentials through JSON config files, environment variables, and CLI flags."
---

Dash loads configuration from three sources, merged in this order (later wins):

1. **Built-in defaults** — sensible starting values
2. **JSON config file** — `config/dash.json` or `dash.json` at the project root
3. **Environment variables** — override specific values (via `.env` or shell)

Credentials can also be loaded from `config/credentials.json` or `credentials.json`, but environment variables always take precedence. For deployed agents, the `--secrets` flag provides a secure alternative (see [CLI flags](#cli-flags)).

For Mission Control users, secrets are stored in an encrypted file — see [encrypted secrets](#encrypted-secrets) below.

## Config file schema

```json
{
  "agents": {
    "<name>": {
      "model": "claude-sonnet-4-20250514",
      "systemPrompt": "You are a helpful assistant.",
      "tools": ["bash", "read_file"],
      "maxTokens": 4096,
      "workspace": "./data/workspace",
      "thinking": { "budgetTokens": 10000 }
    }
  },
  "sessions": {
    "dir": "./data/sessions"
  },
  "logging": {
    "level": "info"
  }
}
```

## Agents

Each key under `agents` defines a named agent configuration.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `model` | `string` | yes | Anthropic model ID (e.g. `claude-sonnet-4-20250514`) |
| `systemPrompt` | `string` | yes | System prompt sent with every request |
| `tools` | `string[]` | no | Tool names to enable. Available: `bash`, `read_file` |
| `maxTokens` | `number` | no | Maximum response tokens (default: `4096`) |
| `workspace` | `string` | no | Working directory for tools, resolved relative to project root |
| `thinking` | `object` | no | Enable extended thinking. See [extended thinking](/extended-thinking) |
| `thinking.budgetTokens` | `number` | — | Token budget for thinking (must be >= 1024, must be < `maxTokens`) |

### Example: multiple agents

```json
{
  "agents": {
    "default": {
      "model": "claude-sonnet-4-20250514",
      "systemPrompt": "You are Dash, a helpful AI assistant.",
      "tools": ["bash", "read_file"],
      "maxTokens": 4096,
      "workspace": "./data/workspace"
    },
    "coder": {
      "model": "claude-sonnet-4-20250514",
      "systemPrompt": "You are Dash Coder, a software engineering assistant.",
      "tools": ["bash", "read_file"],
      "maxTokens": 16000,
      "thinking": { "budgetTokens": 10000 },
      "workspace": "./data/coder-workspace"
    }
  }
}
```

### Per-agent files (`agents/` directory)

Instead of defining all agents in `dash.json`, you can create individual JSON files in an `agents/` directory. Each filename (minus `.json`) becomes the agent name.

```
config/
├── dash.json          # sessions, logging (no agents key needed)
└── agents/
    ├── default.json   # → agent named "default"
    └── coder.json     # → agent named "coder"
```

Each agent file contains just the agent config:

```json
{
  "model": "claude-sonnet-4-20250514",
  "systemPrompt": "You are Dash Coder.",
  "tools": ["bash", "read_file"],
  "maxTokens": 16000
}
```

When an `agents/` directory is present, it **replaces** the `agents` key in `dash.json` entirely. This makes agent definitions portable and easy to share without exposing secrets or session config.

## Sessions

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `dir` | `string` | `./data/sessions` | Directory for session JSONL files |

## Logging

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `level` | `string` | `info` | Log level (e.g. `debug`, `info`, `warn`, `error`) |

## Defaults

When no config file is found, Dash uses these defaults:

```json
{
  "agents": {
    "default": {
      "model": "claude-sonnet-4-20250514",
      "systemPrompt": "You are Dash, a helpful AI assistant. You can use tools to help accomplish tasks.",
      "tools": ["bash", "read_file"],
      "maxTokens": 4096,
      "workspace": "./data/workspace"
    }
  },
  "sessions": { "dir": "./data/sessions" },
  "logging": { "level": "info" }
}
```

## Environment variables

| Variable | Overrides | Description |
|----------|-----------|-------------|
| `ANTHROPIC_API_KEY` | `credentials.json → anthropic.apiKey` | Anthropic API key (required) |
| `MANAGEMENT_API_TOKEN` | — | Enables the Management API when set |
| `MANAGEMENT_API_PORT` | — | Management API port (default: `9100`) |
| `CHAT_API_TOKEN` | — | Enables the Chat API when set |
| `CHAT_API_PORT` | — | Chat API port (default: `9101`) |
| `LOG_LEVEL` | `logging.level` | Log level override |

## CLI flags

The agent server accepts `--config` and `--secrets` flags for deployment scenarios where environment variables and project-relative config files aren't practical.

```bash
node apps/dash/dist/index.js --config /path/to/dash.json --secrets /path/to/secrets.json
```

### `--config <path>`

Load configuration from an explicit path. This can be either:

- **A file** — load that JSON file directly (same as before). If an `agents/` sibling directory exists, agent definitions are loaded from it.
- **A directory** — load `dash.json` from the directory for non-agent settings, and load individual agent definitions from the `agents/` subdirectory.

When `--config` is omitted, Dash searches for `config/dash.json` or `dash.json` at the project root, and additionally checks `config/agents/` for per-agent files.

### `--secrets <path>`

Load secrets from a JSON file. The file is read once and then **deleted** immediately. This is designed for deployment systems that write a temporary secrets file for the agent to consume.

```json
{
  "anthropicApiKey": "sk-ant-...",
  "managementToken": "mgmt-...",
  "chatToken": "chat-..."
}
```

When a secrets file is provided, it takes precedence over environment variables and `credentials.json`.

## Encrypted secrets

Mission Control (desktop app and CLI) stores secrets in an AES-256-GCM encrypted file at `~/.mission-control/secrets.enc`. This replaces plaintext storage and protects API keys, tokens, and credentials at rest.

### How it works

1. On first use, you create an encryption password (minimum 8 characters)
2. A 256-bit key is derived from your password using scrypt
3. Secrets are encrypted and written to `secrets.enc` with 0600 file permissions
4. The derived key is cached in your OS keychain (macOS Keychain or Linux `secret-tool`) so you don't need to re-enter your password on every command

### CLI commands

```bash
mc secrets list                  # List all secret key names
mc secrets get <key>             # Show a masked secret value
mc secrets get <key> --reveal    # Show the full value (for scripting)
mc secrets set <key>             # Store a secret (prompts for value)
mc secrets set <key> --value v   # Store a secret inline
mc secrets delete <key>          # Remove a secret
mc secrets change-password       # Re-encrypt with a new password
mc unlock                        # Unlock the store (cache key in keychain)
mc lock                          # Clear the cached key from the keychain
```

### Migration from plaintext

If you have an existing `~/.mission-control/secrets.json` from before encryption was added, it's migrated automatically. The first command that accesses secrets will prompt you to create a password, encrypt the existing data, and delete the plaintext file.

### Credential precedence

When Mission Control deploys an agent, secrets are injected via a temporary file (read once, then deleted). The agent server resolves credentials in this order:

1. `--secrets` file (highest priority)
2. Environment variables (`.env` or shell)
3. `config/credentials.json`

## Deep merge behavior

<Warning>
  JSON config is deep-merged with defaults. Objects are merged recursively, but arrays are **replaced**, not concatenated. Specifying `tools: ["bash"]` in your config replaces the default `["bash", "read_file"]` entirely.
</Warning>
