---
title: "Management API"
description: "HTTP API for monitoring and controlling a running Dash gateway."
---

The Management API lets you monitor and control the gateway programmatically. It's the same API Mission Control uses to deploy agents, connect messaging apps, and store credentials — useful for automation, scripting, and managing a standalone gateway.

The gateway serves this API on port `9300` by default and binds to `127.0.0.1`, so it isn't reachable from other machines without a reverse proxy or tunnel.

## Authentication

Every endpoint except `GET /health` requires a Bearer token — the gateway's management token (set with `--token`, or generated by Mission Control and stored in the OS keychain). Requests with a missing or invalid token return `401`.

```
Authorization: Bearer <your-management-token>
```

## Health

### GET /health

Returns the gateway's status. **No authentication required** — used for liveness checks.

```bash
curl http://localhost:9300/health
```

**Response**

```json
{
  "status": "healthy",
  "startedAt": "2026-06-21T12:00:00.000Z",
  "pid": 12345,
  "agents": 2,
  "channels": 1
}
```

| Field | Type | Description |
|-------|------|-------------|
| `status` | `"healthy"` | Always `"healthy"` when the gateway is up |
| `startedAt` | `string` | ISO timestamp of when the gateway started |
| `pid` | `number` | Process ID of the gateway |
| `agents` | `number` | Number of registered agents |
| `channels` | `number` | Number of configured messaging channels |

## Agents

Create and manage agents. New and updated agents are persisted to `~/.dash/gateway/agents.json`.

| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/agents` | List all agents |
| `POST` | `/agents` | Create an agent (model, system prompt, tools, workspace, …) |
| `GET` | `/agents/:id` | Get one agent |
| `PUT` | `/agents/:id` | Update an agent's configuration |
| `DELETE` | `/agents/:id` | Delete an agent and clean up its data |
| `POST` | `/agents/:id/enable` | Enable a disabled agent |
| `POST` | `/agents/:id/disable` | Disable an agent without deleting it |

Agent skills are managed under the same prefix — `GET/POST /agents/:id/skills`, `GET/PUT/DELETE /agents/:id/skills/:name`, `POST /agents/:id/skills/install`, and `GET/PATCH /agents/:id/skills/config`. See [Skills](/skills).

**Example: create an agent**

```bash
curl -X POST http://localhost:9300/agents \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "assistant",
    "model": "anthropic/claude-sonnet-4-20250514",
    "systemPrompt": "You are a helpful assistant.",
    "tools": ["bash", "read", "write"]
  }'
```

## Channels

Connect and manage messaging apps (Telegram, WhatsApp). Channels are persisted to `~/.dash/gateway/channels.json`.

| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/channels` | List all channels |
| `POST` | `/channels` | Create a channel and connect its adapter |
| `GET` | `/channels/:name` | Get one channel |
| `PUT` | `/channels/:name` | Update routing rules or access lists |
| `DELETE` | `/channels/:name` | Remove a channel and disconnect its adapter |

See [Messaging Apps](/messaging-apps) for routing rules and access control.

## Credentials

Store provider API keys and messaging tokens in the gateway's encrypted store (`~/.dash/gateway/credentials.enc`). See [Secrets](/secrets).

| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/credentials` | List stored credential key names (values are never returned) |
| `POST` | `/credentials` | Set one or more credentials |
| `DELETE` | `/credentials/:key` | Remove a credential |

## Models

| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/models` | List available models, filtered by which providers you've added credentials for |

The model list is fetched from each provider's API and cached. Adding or changing a credential refreshes it on the next request. This is what populates the model dropdown in Mission Control's deploy wizard. See [AI Providers](/ai-providers).

## Events

| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/events` | Recent gateway events |
| `GET` | `/agents/:agentId/conversations/:conversationId/events` | Replay a conversation's streamed events |

The gateway keeps a durable event log so clients can replay missed chat events after a reconnect. See [Architecture](/architecture#event-log-and-chat-replay).

## Error responses

Failures return a JSON body with an `error` field:

```json
{ "error": "Unauthorized" }
```

| HTTP status | Meaning |
|------------|---------|
| `401` | Missing or invalid `Authorization` header |
| `404` | Agent, channel, or credential not found |
| `400` | Invalid request body |

<Note>
  The Management API (port 9300) and the channel server (port 9200) use separate tokens and separate ports. A management token cannot be used on the channel port, and vice versa.
</Note>
