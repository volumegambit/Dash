Dash provides several ways to interact with agents: the TUI for local terminal use, the gateway for channel routing (Telegram, Mission Control chat), and the Chat API for direct programmatic access over WebSocket.

## TUI (terminal)

The TUI is a terminal-based REPL that connects directly to an agent.

### Running

```bash
npm run tui
```

### Features

- **Streaming text output** — tokens appear as they arrive
- **Spinner with status** — shows "thinking" during inference, "thinking deeply" during extended thinking
- **Tool execution display** — shows tool name, input, and result in a formatted block (truncated to 6 lines)
- **Token usage** — displayed after each response (input/output counts)
- **Color support** — ANSI colors with `NO_COLOR` environment variable respected

### Session handling

Each TUI session creates a conversation ID in the format `cli:<timestamp>`. Sessions are persisted to the JSONL session store but currently cannot be resumed — each TUI launch starts a new conversation.

## Gateway

The gateway is a separate process that owns all channel connections and routes messages to agents via the Chat API. Mission Control's chat panel connects through the gateway — MC chat is just another channel.

### Running

```bash
npm run gateway -- --config gateway.json
```

### Configuration

Create a `gateway.json` file that maps channels to agents:

```json
{
  "channels": {
    "telegram": {
      "adapter": "telegram",
      "token": "your-telegram-bot-token",
      "allowedUsers": ["@yourusername"],
      "agent": "default"
    },
    "mission-control": {
      "adapter": "mission-control",
      "port": 9200,
      "agent": "default"
    }
  },
  "agents": {
    "default": {
      "url": "ws://localhost:9101/ws",
      "token": "your-chat-api-token"
    }
  }
}
```

Each channel specifies an `adapter` type and which `agent` to route messages to. The `agents` section defines how to connect to agent servers via the Chat API.

### Supported adapters

| Adapter | Description | Config fields |
|---------|-------------|---------------|
| `telegram` | Telegram bot via grammY | `token` (required), `allowedUsers` (optional) |
| `mission-control` | WebSocket server for Mission Control chat | `port` (default: 9200) |

### Secrets

When deploying through Mission Control, gateway tokens are stored in the encrypted secret store and injected automatically. See [encrypted secrets](/configuration#encrypted-secrets).

For manual deployments, use `--secrets` to pass tokens securely via a temporary file (deleted after reading):

```bash
npm run gateway -- --config gateway.json --secrets secrets.json
```

The secrets file can override agent tokens and channel tokens:

```json
{
  "agents": { "default": { "token": "secret-chat-token" } },
  "channels": { "telegram": { "token": "secret-bot-token" } }
}
```

## Chat API (WebSocket)

The agent server exposes a WebSocket endpoint for real-time agent interaction. The gateway uses this API to route channel messages to agents, but you can also connect directly. Enable it by setting `CHAT_API_TOKEN` in your environment or via the `--secrets` flag.

### Connecting

```
ws://localhost:9101/ws?token=<your-chat-token>
```

Authentication is via the `token` query parameter on the WebSocket upgrade request. Connections without a valid token are closed with code `4001`.

### Sending a message

Send a JSON message to start a conversation with an agent:

```json
{
  "type": "message",
  "id": "unique-request-id",
  "agent": "default",
  "channelId": "my-app",
  "conversationId": "conv-123",
  "text": "Hello, what can you help me with?"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `type` | `"message"` | Message type |
| `id` | `string` | Unique ID for correlating responses |
| `agent` | `string` | Name of the agent (must match a key in the `agents` config) |
| `channelId` | `string` | Channel identifier for session grouping |
| `conversationId` | `string` | Conversation identifier for session continuity |
| `text` | `string` | The user's message |

### Receiving events

The server streams events back as JSON messages:

```json
{ "type": "event", "id": "unique-request-id", "event": { "type": "text_delta", "text": "Hello" } }
{ "type": "event", "id": "unique-request-id", "event": { "type": "response", "content": "Hello! ...", "usage": { "inputTokens": 10, "outputTokens": 50 } } }
{ "type": "done", "id": "unique-request-id" }
```

| Server message | Description |
|----------------|-------------|
| `event` | Contains an `AgentEvent` — `text_delta`, `thinking_delta`, `tool_use_start`, `tool_result`, `response`, or `error` |
| `done` | Indicates the stream for this request is complete |
| `error` | An error occurred (contains `error` string) |

The `id` field matches your request, allowing multiple concurrent conversations on a single connection.

### Cancelling a request

Send a cancel message to stop an in-flight request:

```json
{
  "type": "cancel",
  "id": "unique-request-id"
}
```

The server acknowledges with a `done` message.

### Message flow

```
Client                          Server
  │                               │
  ├── connect ws://.../ws?token= ─→│
  │                               │
  ├── { type: "message", ... }  ──→│
  │                               ├── resolve agent
  │                               ├── agent.chat()
  │←── { type: "event", ... }  ───┤  (streaming)
  │←── { type: "event", ... }  ───┤
  │←── { type: "event", ... }  ───┤
  │←── { type: "done", ... }   ───┤
  │                               │
```

<Note>
  The Chat API and Management API use separate tokens and run on separate ports. A management token cannot be used on the chat port, and vice versa.
</Note>
